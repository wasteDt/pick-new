# Pick Quote 项目面试问答

## 1. 文档用途

这份文档用于准备 `Pick Quote` 项目的前端面试表达，重点覆盖：

- 项目介绍
- 架构设计
- 技术选型
- 数据流
- 本地存储
- 云同步
- 状态管理
- 性能优化
- 可维护性
- 后续演进方案

文档中的回答不是唯一标准答案，但尽量贴近当前项目的真实实现。

---

## 2. 项目一句话介绍

### 问：请你先简单介绍一下这个项目。

### 答：

这是一个基于 `Plasmo + React + TypeScript + MUI` 实现的浏览器扩展，主要解决用户在浏览网页时快速采集和管理内容的问题，支持保存文本、图片、链接和页面截图。项目采用“本地优先”的设计，主存储放在 `IndexedDB`，同时可选接入 `Convex` 做多设备同步。整体上我重点负责的是采集链路、数据分层、本地存储和同步方案设计。

---

## 3. 项目整体架构

### 问：如果让你用 3 到 5 分钟介绍这个项目的整体架构，你会怎么讲？

### 答：

这个项目是一个浏览器扩展，而不是普通 Web 应用，因为它的核心诉求是在用户浏览任意网页时低打扰地采集内容。扩展能直接接入右键菜单、内容脚本和截图能力，更适合这个场景。

架构上我主要拆成几层：

1. `content script`  
   运行在网页上下文里，负责采集用户选中的文本、提取上下文、生成定位信息。

2. `background`  
   负责注册右键菜单、接收内容脚本消息、调用浏览器 API 完成截图，并统一进入保存链路。

3. `options` 管理页  
   负责展示条目列表、搜索、筛选、删除、导出和查看详情。

4. 数据层  
   我又拆成了 `repository`、`database`、`sync` 三层：
   - `repository` 负责业务编排
   - `database` 负责本地 IndexedDB 的封装
   - `sync` 负责与 Convex 做同步

这样设计的核心是“本地优先、职责解耦、同步可选”。用户操作先落本地，保证性能和离线能力，再异步做云同步增强。

---

## 4. 为什么做成浏览器扩展

### 问：为什么这个项目要做成浏览器扩展，而不是普通 Web 应用？

### 答：

因为这个项目的核心需求不是“展示内容”，而是“在浏览网页时即时采集内容”。如果做成普通网站，用户还要手动复制、切换页面、再粘贴，链路比较长；做成扩展后，可以直接接入：

- 右键菜单
- 内容脚本
- 页面截图能力
- 扩展图标入口

所以扩展形态天然更适合这个场景。

---

## 5. 为什么要有 content script 和 background

### 问：为什么项目里要同时存在 `content script` 和 `background`？

### 答：

它们的职责边界不一样。

- `content script` 更靠近页面，适合做采集，比如读取用户当前选中的文本、提取上下文、获取页面标题和 DOM 锚点。
- `background` 更适合做统一入口和能力调度，比如注册右键菜单、接收采集消息、调用截图 API、把不同来源的数据整理成统一的 `Item`。

如果让 `content script` 直接写库、直接做同步，会导致采集逻辑、存储逻辑和同步逻辑耦合在一起，后续维护会很差。现在这样拆后，采集层只管拿数据，后台只管收口，业务和存储再继续往下沉。

---

## 6. 为什么数据层拆成 repository / database / sync

### 问：为什么不直接在页面里调用 IndexedDB 和 Convex，而是拆成三层？

### 答：

如果页面直接调用 IndexedDB 和 Convex，UI 层就会同时关心展示、存储细节和同步细节，职责会很混乱。

这个项目里保存一条数据并不只是简单写库，还包括：

- 补充 `deviceId`
- 设置 `syncStatus`
- 计算 `hash`
- 做去重
- 决定是否触发云同步

这些都属于业务逻辑，不应该散落在页面组件里。所以我做了三层拆分：

- `database`：只负责 IndexedDB 的表结构、索引和 CRUD
- `sync`：只负责和 Convex 通信
- `repository`：负责把本地存储和云同步串起来

这样做的好处是：

- UI 更干净
- 逻辑集中
- 底层存储方案更容易替换
- 同步策略调整时影响更可控

---

## 7. 为什么选 IndexedDB

### 问：为什么选择 IndexedDB，而不是 localStorage、chrome.storage.local，或者直接全量存到 Convex？

### 答：

我选 IndexedDB 主要有四个原因：

1. 数据体积  
   这个项目不只存文本，还会存图片和截图，`localStorage` 容量太小，也不适合存这类内容。

2. 数据结构  
   项目数据不是简单 key-value，而是有 `type`、`createdAt`、`sourceSite`、`hash`、`syncStatus`、`remoteId` 等字段，还要支持搜索、过滤、排序和同步状态查询，IndexedDB 更适合做结构化存储。

3. 查询能力  
   IndexedDB 支持对象仓库和索引，比 `localStorage` 和 `chrome.storage.local` 更适合主数据场景。

4. 离线能力  
   我不希望把主数据完全交给 Convex，因为这个扩展即使没网也应该正常保存和查看，所以本地主存储放在 IndexedDB，Convex 只作为多设备同步增强。

补充一点，`chrome.storage.local` 在这个项目里也有用，但主要用来保存轻量配置，比如：

- `deviceId`
- `userId`
- `lastPulledAt`

而不是主条目数据。

## 8. Item 类型为什么这样设计

### 问：`Item` 类型里为什么要有 `context`、`hash`、`syncStatus`、`remoteId`、`sourceSite`、`deviceId` 这些字段？

### 答：

这些字段不是简单堆出来的，而是分别为采集、展示、去重和同步服务。

- `context`：解决“只看一句摘录不知道原语境”的问题，提升回看时的可读性
- `hash`：解决重复采集和多设备重复同步问题，用来做去重和幂等
- `syncStatus`：表示本地记录当前是待同步、已同步还是同步失败，便于重试和状态管理
- `remoteId`：映射本地记录和云端记录的关系，因为本地 `id` 和云端 `_id` 不是同一套
- `sourceSite`：为了更高效地做索引和筛选，不用每次查询都重新解析 URL
- `deviceId`：标识这条数据最初来自哪台设备，对同步和排查问题有帮助

所以我设计这些字段时，核心考虑的是四个问题：

- 内容怎么展示
- 数据怎么检索
- 重复内容怎么处理
- 多设备怎么同步

## 9. 保存一条内容的完整链路

### 问：如果用户在网页里选中一段文字并按下快捷键，数据是怎么一步步进入本地库并可能进入云端的？

### 答：

完整链路可以分五步：

1. `content script` 监听快捷键  
   用户按下 `Ctrl + Shift + S` 后，内容脚本读取当前选中的文本，并提取上下文和定位信息。

2. 通过消息发给 `background`  
   内容脚本用 `chrome.runtime.sendMessage` 发送 `capture` 消息。

3. `background` 构造统一条目  
   后台接收到消息后，统一补上本地 `id` 和 `createdAt`，整理成 `Item`。

4. 调用 `repository.saveCapturedItem`  
   仓库层读取同步配置，补充 `deviceId`、`updatedAt`、`syncStatus` 等业务字段。

5. 本地落库并异步同步  
   数据库层会计算 `hash`、补齐 `sourceSite`、执行去重，然后写入 IndexedDB。  
   如果开启了 Convex，同步层再异步把这条数据推到云端，成功后回写 `remoteId` 和同步状态。

这个设计的关键是：**先保证本地保存成功，再做云同步增强。**

---

## 10. 本地搜索是怎么做的

### 问：这个项目里本地搜索是怎么做的？如果数据量到几万条会怎样？

### 答：

当前本地搜索是通过 IndexedDB 做基础索引，再在读取过程中进行过滤。

搜索条件包括：

- `keyword`
- `type`
- `site`
- 时间范围
- `categoryId`

当前实现更接近“按 `createdAt` 倒序遍历，再在内存中根据条件筛选”。  
如果数据量从几百条涨到几万条，会出现两个主要问题：

1. 查询层问题  
   过滤不完全依赖索引，遍历成本会增大。

2. 渲染层问题  
   管理页会渲染更多列表项，滚动和首屏性能会变差。

优化思路我会分两层做：

- 查询层：优化索引设计、减少全量遍历
- 渲染层：做分页、虚拟列表或分段加载

注意这里不能只说“懒加载”，因为懒加载主要优化资源加载，不是本地查询本身的核心手段。

---

## 11. 为什么没有引入 Redux / Zustand

### 问：为什么这个项目没有引入 Redux、Zustand 这类状态管理工具？

### 答：

主要是因为当前状态复杂度还不够高，用 React 自身状态就能覆盖。

当前页面里的状态大多是局部状态，比如：

- `items`
- 搜索关键字
- 类型筛选
- 当前打开的详情弹窗

这些状态基本都只在 `options` 页内部消费，没有太多跨页面共享的需求。

另外，这个项目的数据真源其实不在前端内存，而在：

- `IndexedDB`
- 可选的 Convex 云端

如果我再额外引入一套全局状态管理，就会变成维护两套数据源，复杂度反而会上升。

如果后续功能继续增长，出现下面这些信号时，我会考虑升级状态管理：

- 多个页面共享同一批业务状态
- 异步状态流转越来越复杂
- 组件层级过深，props 透传明显增多
- 需要更强的状态调试和可追踪性

那个阶段我会优先考虑 `Zustand` 这类轻量方案。

## 12. 为什么 Convex 只做可选同步

### 问：为什么这个项目不是直接把所有数据都放到 Convex，而是做成“本地优先 + 可选同步”？

### 答：

因为这是一个浏览器扩展，用户最核心的诉求是：

- 保存动作快
- 离线可用
- 即使没有后端也能工作

所以我的设计是：

- `IndexedDB` 作为主存储
- `Convex` 作为多设备共享增强能力

这样做的好处是：

- 没网也能正常用
- 用户操作链路更短
- 后端服务异常不会影响本地使用
- 云同步可以逐步增强，而不是一开始就把整个项目绑死在后端上

---

## 13. 当前多设备同步的限制

### 问：这个项目的多设备同步目前有什么限制？

### 答：

当前实现能支持基础的新增和更新同步，但还不属于“严肃多端一致性”。

主要限制有：

1. 删除传播不完整  
   当前设备删除数据后，其他设备不一定能同步感知删除，因为当前同步模型本质上还是“拉当前存在的数据”，没有完整删除事件流。

2. 图片和截图没有完整上云  
   当前更偏向同步文本和链接。

3. 没有完整认证体系  
   当前 `userId` 主要来自配置，不属于正式账号体系。

4. 冲突策略较轻  
   主要依赖 `updatedAt` 做最后写入覆盖，还不是严肃的版本控制模型。

---

## 14. 删除不同步的原因

### 问：为什么设备 B 删除了一条数据后，设备 A 不会自动删掉？

### 答：

不是因为 A 本地存了数据就一定删不掉，而是因为当前同步模型没有把“删除”设计成一个可传播的同步事件。

当前流程是：

- B 本地删除
- B 同时删云端
- A 以后再同步时，只会拉“当前还存在的远端记录”

问题就在于：  
被删除的记录已经不存在于云端了，所以 A 根本收不到“它被删了”这个信息。

所以本质上不是本地存储的问题，而是同步模型只支持新增/更新，没有完整支持删除传播。

---

## 15. 如果要做严肃多端一致性怎么升级

### 问：如果要把这个项目升级成严肃多端一致性，你会怎么做？

### 答：

我不会继续只依赖 `updatedAt` 拉主表增量，而是会把同步模型升级成“变更日志 + 服务端版本号”。

核心思路是：

- 主表保存当前数据快照
- 另建一张变更日志表
- 所有新增、更新、删除都写成可重放的 change event
- 客户端不再记录 `lastPulledAt`，而是记录 `lastAppliedSeq`
- 同步时按服务端递增序列拉增量

这样才能解决：

- 删除传播
- 长期离线设备补历史
- 数据复活
- 冲突检测

如果日志区间已经过期，还要支持“全量重建本地状态”。

## 16. 如果后续图片和截图也要上云，怎么设计

### 问：如果后续 `image` 和 `snapshot` 也要上云，你会怎么设计？

### 答：

我不会直接把大 base64 长期塞进条目文档里，而是会把图片类数据拆成两层：

1. 元数据  
   存在 `items` 表里，例如：
   - `type`
   - `source`
   - `hash`
   - `storageId`
   - `mimeType`
   - `width`
   - `height`
   - `size`

2. 二进制资源  
   放到对象存储或 Convex Storage 里。

同步流程会变成：

1. 本地先保存
2. 同步时先上传资源文件
3. 拿到 `storageId`
4. 再同步条目元数据
5. 其他设备根据 `storageId` 拉取或展示图片

这样设计的原因是：

- 避免超大文档
- 提高同步稳定性
- 后续删除更容易做资源回收
- 更适合严肃的多端同步演进

## 17. 性能优化可以怎么讲

### 问：这个项目你做过或考虑过哪些性能优化？

### 答：

当前项目规模不大，但我有几个明确的优化思路：

1. 本地优先  
   先写 IndexedDB，再异步同步，保证用户操作即时反馈。

2. 数据去重  
   通过 `hash` 避免重复入库和重复同步。

3. 索引设计  
   在 IndexedDB 上给高频字段建立索引，降低查询成本。

4. 列表渲染优化  
   如果数据继续增大，会考虑分页或虚拟滚动。

5. 图片同步优化  
   如果未来图片截图上云，会做压缩、缩略图和上传体积控制。

答这类问题时，不要只说“懒加载”，要说清楚是“查询性能”“渲染性能”还是“网络性能”。

## 18. 可维护性怎么讲

### 问：你在这个项目里是怎么保证代码可维护性的？

### 答：

我主要从三方面考虑：

1. 分层清晰  
   采集、后台、业务、存储、同步分开，职责边界比较明确。

2. 类型先行  
   核心 `Item`、`SourceMeta`、`SearchQuery` 都有类型约束，减少字段滥用和调用歧义。

3. 业务入口统一  
   保存、删除、同步这些动作都走 `repository` 层，而不是让各个组件和上下文各自实现一套。

这样后面不管是改存储、改同步还是加新入口，影响面都更可控。

## 22. 面试时可直接使用的项目介绍

这是一个基于 `Plasmo` 开发的浏览器扩展，主要解决用户在浏览网页时快速采集和管理内容的问题，支持保存文本、图片、链接和页面截图。技术栈上我用了 `React + TypeScript + MUI`，数据层采用本地优先的设计，主存储放在 `IndexedDB`，可选接入 `Convex` 做多设备同步。

架构上我把它拆成了几个层次：`content script` 负责在网页里采集内容和上下文，`background` 负责统一处理右键菜单、消息通信和截图能力，`options` 页负责管理界面展示、搜索、筛选和导出。数据层我又拆成了 `repository`、`database`、`sync` 三层，分别负责业务编排、本地持久化和云同步。

我在这个项目里重点考虑的是两个问题：第一是本地优先，保证用户保存动作快、离线也能使用；第二是数据结构和同步设计，比如通过 `hash` 做去重，通过 `syncStatus` 和 `remoteId` 管理同步状态，通过 `context` 保留内容语境。整体上这个项目不只是一个前端页面，更偏向一个有完整采集链路、存储模型和同步机制的扩展工程。
