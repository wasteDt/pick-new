# Pick Quote 项目八股知识点清单

## 1. 文档用途

这份文档用于从八股知识点角度复盘 `Pick Quote` 项目。  
重点不是逐行解释代码，而是把这个项目背后会被问到的前端基础、工程设计和同步设计知识点整理出来。

适用场景：

- 面试前快速复习
- 项目讲解前查漏补缺
- 针对某个模块反推相关理论知识

## 2. 浏览器扩展架构

### 2.1 什么是浏览器扩展

浏览器扩展是一种运行在浏览器环境里的应用形态，可以调用浏览器提供的扩展 API，例如：

- 右键菜单
- 标签页操作
- 内容脚本注入
- 页面截图
- 本地存储

### 2.2 浏览器扩展和普通 Web 应用的区别

- 普通 Web 应用主要运行在页面上下文
- 浏览器扩展通常包含多个运行上下文
- 扩展可以直接调用浏览器能力
- 扩展更适合“页面内采集”和“浏览器级交互”

### 2.3 content script 的职责

`content script` 运行在网页上下文中，适合做：

- 获取选中文本
- 提取 DOM 信息
- 读取当前页面标题、URL、上下文
- 响应页面内快捷键

关键词：

- 靠近页面
- 负责采集
- 不适合承担复杂存储和同步逻辑

### 2.4 background 的职责

`background` 适合做：

- 注册右键菜单
- 接收消息
- 统一调度数据入口
- 调用截图等浏览器 API
- 承接多个入口的数据流

关键词：

- 统一入口
- 调度中心
- 收口不同来源的数据

### 2.5 content script 和 background 通信

常见方式：

- `chrome.runtime.sendMessage`
- `chrome.runtime.onMessage.addListener`

面试要点：

- 为什么要跨上下文通信
- 为什么采集和调度要分开

## 3. 前端架构分层

### 3.1 为什么要分层

分层的本质是：

- 降低耦合
- 提高可维护性
- 降低修改影响面
- 方便后续替换底层实现

### 3.2 这个项目的分层

- 采集层：`content-scripts`
- 后台层：`background`
- 业务层：`repository`
- 存储层：`database`
- 同步层：`sync`
- 展示层：`options` 和 `components`

### 3.3 repository 模式

`repository` 层不是简单转发函数，它负责业务编排，例如：

- 先本地落库
- 再触发同步
- 删除时同时处理本地和云端
- 统一封装保存/删除/搜索入口

关键词：

- 业务编排
- 屏蔽底层实现
- 统一业务入口

### 3.4 高内聚低耦合

理解方式：

- 同一层只做自己该做的事
- UI 不关心数据库细节
- 数据库层不关心 UI 展示
- 同步层不关心页面结构

## 4. React 相关知识点

### 4.1 组件拆分

这个项目可对应的知识点：

- 页面容器组件和展示组件拆分
- 详情弹窗拆分
- 导出相关隐藏组件拆分

关键词：

- 单一职责原则
- 组件复用
- 降低页面复杂度

### 4.2 状态管理为什么不用 Redux / Zustand

当前项目主要状态是：

- 列表数据
- 搜索条件
- 类型筛选
- 当前选中的详情项

这些状态以局部状态为主，因此：

- `useState`
- `useEffect`

就能覆盖。

面试要点：

- 不是不会用全局状态，而是当前复杂度没必要
- 数据真源在 `IndexedDB` 和云端，不在 React 内存 store

### 4.3 什么时候需要升级状态管理

以下信号出现时考虑升级：

- 多个页面共享同一批复杂业务状态
- 跨层级 props 透传严重
- 同步任务状态复杂
- 需要更强的调试和状态追踪能力

### 4.4 React 性能基础

本项目可引出的性能点：

- 列表渲染优化
- 大量数据时的虚拟列表
- 搜索输入的防抖
- 组件重渲染控制

## 6. 本地存储知识点

### 6.1 localStorage

特点：

- 同步 API
- 适合少量字符串配置
- 容量较小
- 不适合复杂查询和大体积数据

### 6.2 chrome.storage.local

特点：

- 浏览器扩展提供的配置存储能力
- 适合轻量配置数据
- 不适合复杂结构化主数据

本项目适合存：

- `deviceId`
- `userId`
- `lastPulledAt`

### 6.3 IndexedDB

特点：

- 异步 API
- 支持对象仓库
- 支持索引
- 支持结构化数据
- 容量更大
- 适合离线本地数据库场景

本项目选择它的原因：

- 需要存截图和图片
- 需要索引和过滤
- 需要离线可用

### 6.4 数据库 schema 设计

本项目的本地表：

- `items`
- `categories`
- `sources`

核心知识点：

- 为什么要分表
- 为什么要建索引
- 如何做版本升级

### 6.5 索引的作用

本项目里常见索引字段：

- `type`
- `createdAt`
- `updatedAt`
- `sourceSite`
- `categoryId`
- `hash`
- `syncStatus`
- `remoteId`

面试要点：

- 索引是为了提升查询效率
- 高频过滤字段适合建索引
- 去重和同步状态也可以靠索引辅助

## 7. 数据去重和幂等

### 7.1 什么是去重

避免同一条内容重复入库。

### 7.2 什么是幂等

多次执行同一个操作，最终结果一致。

### 7.3 本项目为什么需要幂等

因为同一条内容可能来自：

- 快捷键保存
- 右键菜单保存
- 多设备同步回放

### 7.4 hash 的作用

本项目里 `hash` 主要用于：

- 本地去重
- 云端幂等 upsert
- 跨设备避免重复写入

关键词：

- `url + content -> SHA-256`

## 8. 同步设计知识点

### 8.1 本地优先架构

含义：

- 用户操作先写本地
- 云同步是增强能力
- 即使没有网络，核心功能也能工作

优点：

- 响应快
- 离线可用
- 用户体验更稳

### 8.2 最终一致性

多设备同步里常见目标不是强一致，而是最终一致：

- 各设备状态可能短时间不同
- 但经过同步后趋于一致

### 8.3 同步状态字段

本项目里：

- `pending`
- `synced`
- `failed`

作用：

- 标识是否待上传
- 支持失败重试
- 支持同步调度

### 8.4 增量同步

当前项目使用的是轻量的增量同步思想：

- 记录 `lastPulledAt`
- 拉取某个时间点之后更新过的数据

局限：

- 依赖时间戳
- 删除传播不完整
- 长期离线容易出问题

### 8.5 删除同步问题

硬删除的缺陷：

- 远端记录被删后，其他设备无法知道“它被删过”
- 会导致已删数据留在本地

### 8.6 软删除 / tombstone

思路：

- 不直接物理删除
- 记录“已删除”状态
- 让其他设备同步到这个删除事件

### 8.7 长期离线设备问题

如果删除标记保留一段时间后被清理，长期离线设备重新上线后可能：

- 看不到历史删除
- 把旧数据重新上传
- 导致数据复活

### 8.8 严肃多端一致性

如果要做更严肃的一致性，需要：

- 变更日志
- 服务端全局递增序列号
- 删除事件流
- 冲突检测
- 日志失效后的全量重建

## 9. Convex 相关知识点

### 9.1 Convex 在这个项目中的角色

它不是主存储，而是：

- 同步后端
- 云端条目存储
- 多设备共享能力

### 9.2 为什么不用 Convex 做唯一主存储

原因：

- 扩展需要离线可用
- 本地保存速度更重要
- 云端更适合做增强而不是唯一依赖

### 9.3 云端 schema

本项目 Convex 侧目前主要是：

- `items` 表
- 用户维度索引
- `hash` 去重索引

### 9.4 Convex 接口设计

典型接口：

- `upsertFromSync`
- `listSince`
- `removeById`

面试要点：

- 公共 query / mutation 的作用
- 云端如何配合本地同步

## 10. 图片和截图上云知识点

### 10.1 为什么不能直接把 base64 全塞进文档

问题：

- 数据量大
- 文档过重
- 查询慢
- 同步成本高
- 删除回收困难

### 10.2 元数据与资源分离

推荐设计：

- 条目表存元数据
- 对象存储存图片本体
- 文档里通过 `storageId` 关联资源

### 10.3 图片上云流程

基本思路：

1. 本地先保存
2. 上传二进制资源
3. 获得 `storageId`
4. 同步条目元数据
5. 其他设备通过 `storageId` 拉取资源

## 11. 性能优化知识点

### 11.1 查询性能

要点：

- 过滤应尽量走索引
- 避免全量扫描
- 区分“查询慢”和“渲染慢”

### 11.2 渲染性能

如果数据量变大，可考虑：

- 分页
- 无限滚动
- 虚拟列表

### 11.3 图片性能

如果图片上云，可考虑：

- 压缩
- 缩略图
- 尺寸限制
- 延迟加载

### 11.4 输入交互性能

搜索框类场景可引出：

- 防抖
- 节流
- 减少频繁查询

## 12. 用户体验知识点

### 12.1 为什么要本地优先

因为用户保存内容时最在意的是：

- 立刻成功
- 不被网络阻塞

### 12.2 为什么保留上下文

因为只保存一句摘录容易丢失语境。  
保留段落和前后文能提升后续回看体验。

### 12.3 为什么做右键菜单和快捷键

目的是缩短操作路径，降低采集成本。

### 12.4 为什么支持导出

导出能力体现：

- 数据主权
- 可迁移性
- 用户信任感
